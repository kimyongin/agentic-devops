# Task Writer

> 파이프라인 규칙, 산출물 체계, 역할, 변경 처리는 `agentic-devops` 스킬을 따른다.

- 승인된 DESIGN을 입력으로 받는다. 산출물 ID: `TASK-xxxx-slug`. 메타데이터에 참조 DESIGN을 필수 기재.
- 증분 원칙: 기능 단위로 TASK를 증분 갱신. 이전 스프린트 TASK는 누적(Living Document)으로 관리.

## 실행 절차

### 1단계: 입력 확인

1. 입력 설계 문서(`DESIGN-xxxx-slug`)를 읽고 내용을 파악한다.
2. DESIGN 상태가 APPROVED인지 확인한다. DRAFT이면 사용자에게 승인 필요를 알린다.
3. 기존 TASK가 있으면 읽고 증분 갱신 여부를 판단한다.

### 2단계: 작업 계획 생성

아래 **작업 계획 원칙**과 **템플릿**을 따라 `docs/ai-devops/TASK/TASK-xxxx-slug.md`를 생성한다.

### 3단계: 맥락 노트 생성

아래 **맥락 노트 템플릿**에 따라 `docs/ai-devops/TASK/CONTEXT-xxxx-slug.md`를 함께 생성한다.

### 4단계: 보고

```
## 작업 계획 보고서
- 생성 문서: TASK-xxxx-slug, CONTEXT-xxxx-slug
- 추가 TASK 필요: (있음 — 제안 목록 / 없음)
- 발견 사항: (설계에서 모호하거나 추가 확인이 필요한 부분)
- 오픈 이슈: (해결되지 않은 질문)
- 판단 근거: (주요 결정과 근거)
- 잔여 리스크: (남아 있는 위험 요소)
```

## 작업 계획 원칙

### 작업 단위로 쪼갠다(WBS)

- 기능/계약/데이터/테스트/관측성/배포를 각각 작업으로 분해한다.
- "나중에 하자" 금지: 특히 관측성과 롤백은 초반에 포함.

### 순서를 명시한다(Dependency & Order)

- 선행 조건이 있는 작업(스키마/계약/CI)은 우선 배치한다.

### 완료 기준을 적는다(Definition of Done)

- 각 작업은 "끝났는지" 판단 가능한 체크가 있어야 한다(테스트 통과, 지표 노출 등).

### 작업 분할 기준(TASK Splitting)

하나의 DESIGN에서 복수의 TASK 문서를 생성할 수 있다. 아래 기준에 따라 분할 여부를 판단한다:

- WBS 항목이 20개를 초과하여 단일 문서로 추적이 어려운 경우
- 기능 영역별로 독립적인 마일스톤과 완료 기준이 필요한 경우
- 단계적 구현이 필요하여 선행 TASK 완료 후 후속 TASK를 계획해야 하는 경우

분할 시 각 TASK는 동일한 DESIGN을 참조하되, 담당 영역을 slug에 반영한다.
(예: TASK-0001-login-api, TASK-0002-user-profile-api)

## 작업 기억 프로토콜

AI는 대화가 길어지거나 새 대화를 시작하면 맥락을 잃는다. 큰 작업 수행 시 반드시 **3종 기억 문서**를 운영한다.

| 문서 | 역할 | 위치 |
|------|------|------|
| 계획서(TASK) | 무엇을 만들 것인지 — 작업 분해/순서/검증 기준 | `docs/ai-devops/TASK/TASK-xxxx-slug.md` |
| 맥락 노트(CONTEXT) | 왜 그렇게 결정했는지 — 결정 근거, 관련 자료 위치, 중간 합의 사항 | `docs/ai-devops/TASK/CONTEXT-xxxx-slug.md` |
| 진행 체크리스트 | 무엇을 끝냈고 무엇이 남았는지 — TASK의 WBS 상태 컬럼으로 관리 | TASK 문서 내 WBS 테이블 |

### 운영 절차

1. **계획 수립**: AI에게 전체 계획을 세우게 한다.
2. **계획 검토**: AI가 세운 계획을 검토하여 핵심을 오해하지 않았는지 확인한다.
3. **문서 저장**: 승인된 계획을 3종 문서(TASK, CONTEXT, WBS 상태)로 저장한다.
4. **새 대화에서 이어서 작업**: 새 대화를 시작할 때 저장된 문서를 읽고 현재 상황을 파악한 뒤 작업을 이어간다.
5. **소단위 작업**: 한두 개 작업만 시키고, 완료 후 WBS 상태를 업데이트한다.
6. **반복**: 5번을 반복하며, 필요 시 맥락 노트에 새 결정사항을 추가한다.
7. **기술 부채 반영**: 코드 리뷰에서 발견된 기술 부채 후보가 있으면 맥락 노트에 기록하고, 다음 TASK 생성 시 참고한다.

## 작업 계획 템플릿

`docs/ai-devops/TASK/TASK-xxxx-slug.md` 형태로 생성한다.

### [TASK-xxxx-slug] 작업 계획 제목

#### 메타데이터
- 문서 ID: `TASK-xxxx-slug`
- 버전: `v0.1.0`
- 참조 설계: `DESIGN-xxxx-slug`
- 작성일: `YYYY-MM-DD`
- 상태: `DRAFT | APPROVED | DEPRECATED`
- 오너(승인자): `@name`

#### 1) 범위 요약(Scope Summary)
- 이번 구현 범위에 포함:
- 이번 구현 범위에 미포함:
- 전제(Assumptions):

#### 2) 산출물(Deliverables)
- 코드:
- 테스트:
- 문서(계약/운영):
- 대시보드/알람:

#### 3) WBS(작업 분해)

> 작업은 가능한 한 0.5~2일 단위로 쪼개고, "검증 방법"을 함께 적는다.

| ID | 작업 | 설명 | 의존성 | 검증(DoD) | 담당 | 상태 |
|---|---|---|---|---|---|---|
| T1 |  |  |  |  |  | TODO / IN_PROGRESS / DONE |
| T2 |  |  |  |  |  | TODO |

#### 4) 마일스톤/일정(Milestones)
- 프로젝트 규모에 맞게 마일스톤을 정의한다. 계약 확정 → 핵심 구현/테스트 → 관측성/배포 → 안정화 순서를 권장.

#### 5) 계약/스키마 변경 계획(Contract & Schema)
- API 변경: 버전 전략, 하위호환 정책
- 이벤트 변경: 멱등성/재처리
- DB/스키마: 마이그레이션, 롤백

#### 6) 테스트 계획(Test Plan)
- DESIGN의 테스트 전략을 기반으로, Unit/Integration/Contract/E2E 각각의 실행 시점과 대상을 확정한다.

#### 7) 운영 준비 체크리스트
- [ ] 배포/롤백 전략 정의 (Canary, 데이터 마이그레이션 타이밍)
- [ ] 관측성 구성 계획 (로그/메트릭/대시보드/알람, SLO 모니터링)
- [ ] 보안/권한/시크릿 관리 계획

#### 8) 리스크 & 대응(Risks)

| 리스크 | 영향 | 가능성 | 대응 |
|---|---|---|---|
|  |  |  |  |

#### 9) 완료 기준(Definition of Done)
- [ ] 모든 자동 게이트(CI) 통과
- [ ] 핵심 유즈케이스 유닛/통합 테스트 통과
- [ ] 계약 테스트 통과(호환성 확인)
- [ ] 관측성(대시보드/알람) 구성 완료
- [ ] 배포/롤백 절차 문서화 및 리허설
- [ ] 배포 후 안정화 확인(SLO 위반 없음, 오류율 정상)
- [ ] 운영 문서 업데이트
