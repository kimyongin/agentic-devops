# Gate Keeper

## 프로세스 규칙

이 프로젝트는 **PRD → DESIGN → TASK → CODE → PR → RELEASE** 파이프라인을 따른다.
모든 단계는 이전 단계의 승인된 산출물을 입력으로 받으며, 승인 없이 다음 단계로 진행할 수 없다.

파이프라인, 산출물 체계, 역할, 문서 버전 관리, 변경 처리 규칙은 `SKILL.md`를 참조한다.
게이트 검증은 모든 산출물 유형(PRD / DESIGN / TASK / PR / RELEASE)에 대해 수행한다.

---

## 실행 절차

### 1단계: 입력 확인

1. 검증 대상 산출물의 유형을 판별한다 (PRD / DESIGN / TASK / 코드·PR / RELEASE).
2. 해당 산출물을 읽는다.

### 2단계: 산출물 검증

산출물 유형에 따라 해당 레퍼런스의 **셀프 체크** 섹션을 검증 기준으로 사용한다.

| 산출물 유형 | 검증 기준 |
|-------------|-----------|
| PRD | `prd-writer.md` 3단계: 셀프 체크 |
| DESIGN | `design-writer.md` 3단계: 셀프 체크 + 검증 게이트 |
| TASK | `task-writer.md` 4단계: 셀프 체크 |
| 코드/PR | `code-writer.md` 3단계: 셀프 체크 + `pr-manager.md` 6단계: 셀프 체크 |
| RELEASE | `release-manager.md` 8단계: 셀프 체크 |

### 3단계: 프로세스 일관성 검증

산출물 내용과 무관하게, 프로세스 준수 여부를 확인한다:

- [ ] 산출물 ID/버전이 부여되었는가?
- [ ] 이전 단계 산출물 ID를 참조하고 있는가? (DESIGN → PRD, TASK → DESIGN)
- [ ] 변경 이력(Changelog)이 기록되었는가?
- [ ] 승인되지 않은 범위가 포함되어 있지 않은가?

### 4단계: 보고

```
## 게이트 검증 보고서
- 검증 대상: (산출물 ID + 유형)
- 검증 기준: (적용한 체크리스트)

### 발견 사항
| 번호 | 심각도 | 항목 | 내용 | 권고 |
|------|--------|------|------|------|
| 1 | Critical/Major/Minor | | | |

### 통과 항목
- (검증을 통과한 주요 항목)

### 판단 근거
- (주요 판단의 이유)

### 잔여 리스크
- (남아 있는 위험 요소)

### 종합 판정
- 게이트 통과 / 수정 후 재검증 / 이전 단계 재작업 필요
```

## 게이트(Gates)

### 수동 게이트(사람 승인)

- 설계 문서 승인: 경계, 계약, 유즈케이스, 리스크, 검증전략
- 작업 계획 승인: WBS, 순서, 테스트, 배포, 롤백, 관측성
- PR 승인: 코드 가이드 준수, 테스트 통과, 계약 변경 여부
- 릴리스 승인: RELEASE 문서 확인, 배포 환경, 롤백 계획

### 자동 게이트(CI)

- 포맷/린트/정적 분석
- 단위 테스트(Unit)
- 통합 테스트(Integration)
- 계약 테스트(Contract; API/이벤트/스키마)
- 아키텍처 규칙 검사(레이어 의존 금지 등)
- 보안 스캐닝(의존성 취약점/시크릿 검사)
- 라이센스 검사
- 빌드/패키징
- 커버리지/품질 기준(선택)

## 프로세스 성과 측정(KPI)

| 지표 | 측정 방법 | 목적 |
|---|---|---|
| 게이트 통과율 | 1차 제출에서 승인된 비율 | AI 산출물 품질 평가 |
| CHANGE_REQUEST 빈도 | 스프린트당 발생 횟수 | 설계/작업 완성도 평가 |
| 배포 후 장애율 | 릴리스 후 24시간 내 장애 수 | 프로세스 안정성 평가 |
| 리드타임 | PRD 등록 → 릴리스 완료까지 소요 시간 | 프로세스 효율성 평가 |
| 롤백 빈도 | 릴리스 대비 롤백 비율 | 배포 품질 평가 |
| 문서-코드 괴리 건수 | 코드 리뷰에서 발견된 문서 불일치 수 | 계약 우선 원칙 준수도 |

분기별로 KPI를 리뷰하고, 프로세스 개선 ADR을 작성한다.

## 핵심 원칙

- 모든 검증 항목을 빠짐없이 확인한다 — 선택적 스킵 금지.
- "통과"와 "미통과"를 모두 명시하여 검증 범위를 투명하게 한다.
- Critical 이슈가 1건이라도 있으면 게이트를 통과시키지 않는다.
- 프로세스 일관성(ID/버전/참조/변경이력)은 내용 검증과 별도로 반드시 확인한다.
