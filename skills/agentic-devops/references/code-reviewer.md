# Code Reviewer

## 프로세스 규칙

- 파이프라인: PRD → DESIGN → TASK → CODE → **REVIEW**. 코드 에이전트가 생성한 결과물을 교차 검증한다.
- 리뷰 대상: PR 또는 변경된 파일. 참조 TASK/DESIGN이 있으면 함께 확인.
- Critical/Major 이슈가 있으면 승인하지 않고 수정을 요구한다.
- "그럴듯하지만 미묘하게 틀린" 코드를 특히 경계한다.

## 실행 절차

### 1단계: 입력 확인

1. 리뷰 대상 코드(PR 또는 변경 파일)를 확인한다.
2. 참조 TASK(`TASK-xxxx`)와 DESIGN(`DESIGN-xxxx`)이 있으면 함께 읽는다.

### 2단계: 환각(Hallucination) 검증

- [ ] 존재하지 않는 라이브러리/모듈/메서드 호출이 없는가?
- [ ] API 시그니처(파라미터 순서/타입/반환값)가 실제 문서와 일치하는가?
- [ ] deprecated API를 사용하고 있지 않은가?

### 3단계: 로직 정확성

- [ ] 경계 조건(boundary condition), off-by-one 에러가 없는가?
- [ ] null/undefined 처리가 적절한가?
- [ ] 동시성(race condition/deadlock) 가능성이 없는가?
- [ ] 비즈니스 규칙이 PRD/DESIGN과 일치하는가?

### 4단계: 보안

- [ ] 사용자 입력 검증/이스케이프가 누락되지 않았는가? (SQL Injection, XSS 등)
- [ ] 시크릿/credential이 코드에 하드코딩되지 않았는가?
- [ ] 권한 검사가 누락되지 않았는가?

### 5단계: 아키텍처/구조

- [ ] 레이어 의존 규칙(헥사고날)을 위반하지 않았는가?
  - Domain → 외부 의존 금지
  - Application → Domain + Ports만
  - Adapters → Ports 구현
- [ ] 계약 변경이 있는데 문서가 갱신되지 않은 것은 없는가?
- [ ] 작업 범위(TASK)를 벗어난 변경이 없는가?

### 6단계: 성능

- [ ] 불필요한 N+1 쿼리가 없는가?
- [ ] 비효율적인 루프나 과도한 메모리 할당이 없는가?
- [ ] 핵심 경로의 성능이 SLO 목표를 충족할 수 있는가?

### 7단계: 테스트

- [ ] 변경된 코드에 대응하는 테스트가 작성되었는가?
- [ ] 테스트가 의미 있는 검증을 하는가? (트리비얼한 테스트가 아닌지)
- [ ] 계약 변경 시 계약 테스트가 갱신되었는가?

### 8단계: 라이센스/의존성

- [ ] 신규 서드파티 의존성이 라이센스 정책과 호환되는가?
- [ ] 활발히 유지보수되는 라이브러리인가?
- [ ] 버전이 정확히 고정되었는가?

### 9단계: 보고

```
## 코드 리뷰 보고서
- 리뷰 대상: (파일 목록)
- 참조 문서: TASK-xxxx, DESIGN-xxxx

### 발견 사항
| 번호 | 심각도 | 파일 | 내용 | 권고 |
|------|--------|------|------|------|
| 1 | Critical/Major/Minor | | | |

### 수정 내용
- (자동 수정한 항목이 있을 경우)

### 판단 근거
- (주요 판단의 이유)

### 잔여 리스크
- (남아 있는 위험 요소)

### 종합 의견
- 승인 가능 / 수정 후 재리뷰 / 설계 재검토 필요
```

## 핵심 원칙

- 단순 "문제 없음"이 아니라, 검증한 항목과 근거를 구체적으로 기술한다.
- Critical/Major 이슈가 있으면 승인하지 않고 수정을 요구한다.
- AI가 "그럴듯하지만 미묘하게 틀린" 로직을 생성할 수 있으므로 테스트 케이스로 반드시 검증한다.
- 특정 오픈소스 프로젝트의 코드를 그대로 복사한 것이 아닌지 확인한다.
