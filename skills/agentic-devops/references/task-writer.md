# Task Writer

## 프로세스 규칙

- 파이프라인: PRD → DESIGN → **TASK** → CODE. 승인된 DESIGN을 입력으로 받는다.
- 산출물 ID: `TASK-xxxx` 형식. 메타데이터에 참조 DESIGN을 필수 기재.
- 버전: 계약/경계/데이터 모델 변경 → Major, 범위/일정 변경 → Minor, 오타/포맷 → Patch.
- APPROVED 문서를 변경하면 DRAFT로 되돌리고 재승인. 승인 범위 이탈 시 `CHANGE_REQUEST` 발행.
- 증분 원칙: 기능 단위로 TASK를 증분 갱신. 이전 스프린트 TASK는 누적(Living Document)으로 관리.

## 실행 절차

### 1단계: 입력 확인

1. 입력 설계 문서(`DESIGN-xxxx`)를 읽고 내용을 파악한다.
2. DESIGN 상태가 APPROVED인지 확인한다. DRAFT이면 사용자에게 승인 필요를 알린다.
3. 기존 TASK가 있으면 읽고 증분 갱신 여부를 판단한다.

### 2단계: 작업 계획 생성

아래 **작업 계획 원칙**과 **템플릿**을 따라 `docs/ai-devops/TASK/TASK-xxxx.md`를 생성한다.

### 3단계: 맥락 노트 생성

아래 **맥락 노트 템플릿**에 따라 `docs/ai-devops/TASK/CONTEXT-xxxx.md`를 함께 생성한다.

### 4단계: 셀프 체크

- [ ] 모든 작업에 검증 방법(DoD)이 있는가?
- [ ] 의존성 순서가 명확한가?
- [ ] 관측성/롤백이 포함되었는가?
- [ ] 테스트 계획의 실행 시점이 마일스톤과 연동되었는가?
- [ ] 계약/스키마 변경이 식별되고 계획에 포함되었는가?
- [ ] 맥락 노트(CONTEXT)가 함께 생성되었는가?

### 5단계: 보고

```
## 작업 계획 보고서
- 생성 문서: TASK-xxxx, CONTEXT-xxxx
- 발견 사항: (설계에서 모호하거나 추가 확인이 필요한 부분)
- 오픈 이슈: (해결되지 않은 질문)
- 판단 근거: (주요 결정과 근거)
- 잔여 리스크: (남아 있는 위험 요소)
```

## 작업 계획 원칙

### 작업 단위로 쪼갠다(WBS)

- 기능/계약/데이터/테스트/관측성/배포를 각각 작업으로 분해한다.
- "나중에 하자" 금지: 특히 관측성과 롤백은 초반에 포함.

### 순서를 명시한다(Dependency & Order)

- 선행 조건이 있는 작업(스키마/계약/CI)은 우선 배치한다.

### 완료 기준을 적는다(Definition of Done)

- 각 작업은 "끝났는지" 판단 가능한 체크가 있어야 한다(테스트 통과, 지표 노출 등).

## 작업 기억 프로토콜

AI는 대화가 길어지거나 새 대화를 시작하면 맥락을 잃는다. 큰 작업 수행 시 반드시 **3종 기억 문서**를 운영한다.

| 문서 | 역할 | 위치 |
|------|------|------|
| 계획서(TASK) | 무엇을 만들 것인지 — 작업 분해/순서/검증 기준 | `docs/ai-devops/TASK/TASK-xxxx.md` |
| 맥락 노트(CONTEXT) | 왜 그렇게 결정했는지 — 결정 근거, 관련 자료 위치, 중간 합의 사항 | `docs/ai-devops/TASK/CONTEXT-xxxx.md` |
| 진행 체크리스트 | 무엇을 끝냈고 무엇이 남았는지 — TASK의 WBS 상태 컬럼으로 관리 | TASK 문서 내 WBS 테이블 |

### 운영 절차

1. **계획 수립**: AI에게 전체 계획을 세우게 한다.
2. **계획 검토**: AI가 세운 계획을 검토하여 핵심을 오해하지 않았는지 확인한다.
3. **문서 저장**: 승인된 계획을 3종 문서(TASK, CONTEXT, WBS 상태)로 저장한다.
4. **새 대화에서 이어서 작업**: 새 대화를 시작할 때 저장된 문서를 읽고 현재 상황을 파악한 뒤 작업을 이어간다.
5. **소단위 작업**: 한두 개 작업만 시키고, 완료 후 WBS 상태를 업데이트한다.
6. **반복**: 5번을 반복하며, 필요 시 맥락 노트에 새 결정사항을 추가한다.

### 핵심 팁

- AI에게 계획이 승인되면 바로 작업을 시작하게 하지 않는다. **"먼저 이 계획을 문서로 저장해"**라고 지시한다.
- 저장 후 **새 대화**를 시작하여 저장된 문서를 읽고 이어서 작업하게 한다.
- 한 번에 모든 작업을 시키지 않고, **한두 개씩 시키며 중간중간 체크리스트를 업데이트**한다.

- **Unit**: 실행 시점 M2, 대상 범위, 커버리지 목표
- **Integration**: 실행 시점 M3, 테스트 환경
- **Contract**: 실행 시점 M3, 검증 대상 계약 목록
- **E2E**: 실행 시점 M4, 대상 사용자 플로우
- **성능/부하**(필요 시): 실행 시점, 목표 TPS/지연

### [CONTEXT-xxxx] 맥락 노트 제목

- 문서 ID: `CONTEXT-xxxx`
- 참조 TASK: `TASK-xxxx`
- 참조 DESIGN: `DESIGN-xxxx`
- 작성일: `YYYY-MM-DD`
- 최종 갱신: `YYYY-MM-DD`

#### 1) 결정 사항(Decisions)

> 작업 중 내린 결정과 그 근거를 기록한다.

| 번호 | 결정 내용 | 근거 | 일자 |
|------|-----------|------|------|
| D1 |  |  |  |

#### 2) 관련 자료(References)

> 참조한 문서, 코드, 외부 자료의 위치를 기록한다.

#### 3) 중간 합의 사항(Agreements)

> 사람과 AI 간, 또는 팀원 간 합의한 내용을 기록한다.
