# Quality Checker

## 프로세스 규칙

이 프로젝트는 **PRD → DESIGN → TASK → CODE** 파이프라인을 따른다.
모든 단계는 이전 단계의 승인된 산출물을 입력으로 받으며, 승인 없이 다음 단계로 진행할 수 없다.

### 산출물 체계(Traceability)

모든 산출물은 **ID/버전**을 가진다. 각 단계는 이전 단계 산출물 ID를 **참조**한다.

- **PRD** — `PRD-xxxx`: 배경, 목표, 사용자 스토리, 성공 기준, 비범위
- **설계 문서** — `DESIGN-xxxx`: 입력 `PRD-xxxx`. 도메인 모델, 유즈케이스, 헥사고날 아키텍처, 계약, 관측성, 테스트 전략
- **작업 계획** — `TASK-xxxx`: 입력 `DESIGN-xxxx`. WBS, 마일스톤, 테스트/배포/롤백, DoD
- **코드/PR** — `PR-xxxx`: 입력 `TASK-xxxx`. 구현 코드, 테스트, 계약 문서 변경
- **릴리스** — `RELEASE-xxxx`: 포함 PR 목록, 변경 요약, 배포 환경, 롤백 계획
- **ADR** — `ADR-xxxx`: 결정 배경, 선택지, 트레이드오프, 결정 사항

### 문서 버전 관리 규칙

| 변경 유형 | 버전 증분 | 예시 |
|---|---|---|
| 계약/경계/데이터 모델 변경 | Major (v1 → v2) | API 엔드포인트 추가, 이벤트 스키마 변경 |
| 범위/일정/전략 변경 | Minor (v1.0 → v1.1) | 마일스톤 조정, 테스트 전략 수정 |
| 오타/포맷/표현 수정 | Patch (v1.1.0 → v1.1.1) | 문구 정정, 링크 수정 |

- APPROVED 상태의 문서를 변경하면 상태를 DRAFT로 되돌리고 재승인.
- 변경 이력(Changelog) 섹션에 버전별 변경 사유를 반드시 기록.

### 역할과 책임

- **Owner(사람)**: PRD 작성/승인, 설계 승인, 작업 계획 승인, 최종 릴리스 승인
- **AI(도구)**: 가이드 기반 산출물 생성, 변경 제안, 체크리스트 자동 점검
- **Reviewer(사람)**: 경계, 계약, 리스크, 테스트 전략 검토, 게이트 통과 여부 판단

### 변경 처리 규칙

승인 범위를 벗어나는 변경은 `CHANGE_REQUEST` 발행.

- **설계 변경**: 도메인 경계/계약/데이터 모델/유즈케이스 변경 → DESIGN 갱신 재승인 → TASK 갱신 → 코드 반영
- **작업 변경**: 작업 범위/순서/마일스톤/전략 변경 → TASK 갱신 재승인 → 코드 반영
- **코드 변경**: 작업 범위 내 구현 수정 → PR 리뷰 및 승인만으로 반영 가능

## 실행 절차

### 1단계: 입력 확인

1. 검증 대상 산출물의 유형을 판별한다 (PRD / DESIGN / TASK / 코드·PR).
2. 해당 산출물을 읽는다.

### 2단계: 산출물 유형별 검증

#### PRD 검증

- [ ] 필수 섹션(배경, 목표/성공 기준, 사용자 스토리, 범위, 제약사항, 변경 이력) 포함
- [ ] 메타데이터(문서 ID, 버전, 상태, 오너) 정확
- [ ] 범위와 비범위가 명확히 구분
- [ ] 비기능 요구사항(NFR)이 정량적으로 기술
- [ ] 사용자 스토리가 한글로 작성

#### DESIGN 검증

- [ ] 도메인 경계가 명확한가?
- [ ] 포트/어댑터가 분리되어 있는가?
- [ ] 계약(API/이벤트/스키마)이 문서화되었는가?
- [ ] 테스트 전략과 관측성이 포함되었는가?
- [ ] SLO/SLI가 정의되었는가?
- [ ] 마이그레이션/롤백이 고려되었는가?
- [ ] 멀티 서비스 변경 시 배포 순서와 호환성 정책이 명시되었는가?
- [ ] 데이터 거버넌스(PII/보존/분류)가 검토되었는가?
- [ ] 리스크/가정/오픈 이슈가 기록되었는가?

#### TASK 검증

- [ ] 모든 작업에 검증 방법(DoD)이 있는가?
- [ ] 의존성 순서가 명확한가?
- [ ] 관측성/롤백이 포함되었는가?
- [ ] 테스트 계획의 실행 시점이 마일스톤과 연동되었는가?
- [ ] 계약/스키마 변경이 식별되고 계획에 포함되었는가?
- [ ] 맥락 노트(CONTEXT)가 함께 생성되었는가?

#### 코드/PR 검증

- [ ] 레이어 의존 규칙 준수
- [ ] 계약 변경 시 문서 동시 수정
- [ ] 테스트 작성 완료
- [ ] 보안 확인
- [ ] AI 생성 코드 리뷰 항목 확인 (환각/로직/보안/라이센스/성능)
- [ ] PR 템플릿 필수 항목 기재

### 3단계: 프로세스 일관성 검증

산출물 내용과 무관하게, 프로세스 준수 여부를 확인한다:

- [ ] 산출물 ID/버전이 부여되었는가?
- [ ] 이전 단계 산출물 ID를 참조하고 있는가? (DESIGN → PRD, TASK → DESIGN)
- [ ] 변경 이력(Changelog)이 기록되었는가?
- [ ] 승인되지 않은 범위가 포함되어 있지 않은가?

### 4단계: 보고

```
## 품질 검증 보고서
- 검증 대상: (산출물 ID + 유형)
- 검증 기준: (적용한 체크리스트)

### 발견 사항
| 번호 | 심각도 | 항목 | 내용 | 권고 |
|------|--------|------|------|------|
| 1 | Critical/Major/Minor | | | |

### 통과 항목
- (검증을 통과한 주요 항목)

### 판단 근거
- (주요 판단의 이유)

### 잔여 리스크
- (남아 있는 위험 요소)

### 종합 판정
- 게이트 통과 / 수정 후 재검증 / 이전 단계 재작업 필요
```

## 게이트(Gates)

### 수동 게이트(사람 승인)

- 설계 문서 승인: 경계, 계약, 유즈케이스, 리스크, 검증전략
- 작업 계획 승인: WBS, 순서, 테스트, 배포, 롤백, 관측성
- PR 승인: 코드 가이드 준수, 테스트 통과, 계약 변경 여부
- 릴리스 승인: RELEASE 문서 확인, 배포 환경, 롤백 계획

### 자동 게이트(CI)

- 포맷/린트/정적 분석
- 단위 테스트(Unit)
- 통합 테스트(Integration)
- 계약 테스트(Contract; API/이벤트/스키마)
- 아키텍처 규칙 검사(레이어 의존 금지 등)
- 보안 스캐닝(의존성 취약점/시크릿 검사)
- 라이센스 검사
- 빌드/패키징
- 커버리지/품질 기준(선택)

## 릴리스 절차

### 릴리스 산출물

- `RELEASE-xxxx`: 포함된 PR 목록, 변경 요약, 배포 환경, 롤백 계획
- 릴리스 노트: 사용자/운영자 관점의 변경 사항 요약(외부 공개용)

### 릴리스 프로세스

1. **릴리스 후보(RC) 확정**: 포함할 PR 목록 확정, 모든 자동 게이트 통과 확인.
2. **배포 전 점검**: 롤백 계획 확인, 관측성(대시보드/알람) 준비.
3. **단계 배포**: Canary/Blue-Green 등 점진적 배포.
4. **배포 후 검증**: 핵심 메트릭(오류율/지연/트래픽) 모니터링, SLO 위반 여부 확인.
5. **릴리스 확정 또는 롤백**: 이상 없으면 확정, 문제 시 즉시 롤백.
6. **사후 정리**: 릴리스 노트 발행, 운영 문서 갱신.

## 프로세스 성과 측정(KPI)

| 지표 | 측정 방법 | 목적 |
|---|---|---|
| 게이트 통과율 | 1차 제출에서 승인된 비율 | AI 산출물 품질 평가 |
| CHANGE_REQUEST 빈도 | 스프린트당 발생 횟수 | 설계/작업 완성도 평가 |
| 배포 후 장애율 | 릴리스 후 24시간 내 장애 수 | 프로세스 안정성 평가 |
| 리드타임 | PRD 등록 → 릴리스 완료까지 소요 시간 | 프로세스 효율성 평가 |
| 롤백 빈도 | 릴리스 대비 롤백 비율 | 배포 품질 평가 |
| 문서-코드 괴리 건수 | 코드 리뷰에서 발견된 문서 불일치 수 | 계약 우선 원칙 준수도 |

분기별로 KPI를 리뷰하고, 프로세스 개선 ADR을 작성한다.

## 핵심 원칙

- 모든 검증 항목을 빠짐없이 확인한다 — 선택적 스킵 금지.
- "통과"와 "미통과"를 모두 명시하여 검증 범위를 투명하게 한다.
- Critical 이슈가 1건이라도 있으면 게이트를 통과시키지 않는다.
- 프로세스 일관성(ID/버전/참조/변경이력)은 내용 검증과 별도로 반드시 확인한다.
