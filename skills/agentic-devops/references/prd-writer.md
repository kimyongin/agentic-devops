# PRD Writer

## 프로세스 규칙

- 파이프라인: **PRD** → DESIGN → TASK → CODE → PR → RELEASE. PRD는 첫 단계 산출물이다.
- 산출물 ID: `PRD-xxxx` 형식. 메타데이터에 문서 ID, 버전, 상태, 오너를 필수 기재.
- 버전: 계약/경계/데이터 모델 변경 → Major, 범위/일정 변경 → Minor, 오타/포맷 → Patch.
- APPROVED 문서를 변경하면 DRAFT로 되돌리고 재승인. 승인 범위 이탈 시 `CHANGE_REQUEST` 발행.
- 증분 원칙: 한 번에 완성하지 않는다. 기능 단위로 증분 갱신하고 승인된 범위만 다음 단계로 진행.

## 실행 절차

### 1단계: 입력 확인

1. 사용자로부터 기능/프로젝트 요구사항을 확인한다.
2. 기존 PRD가 있으면 읽고 증분 갱신 여부를 판단한다.

### 2단계: PRD 문서 생성

아래 **템플릿**의 모든 섹션을 포함하여 `docs/ai-devops/PRD/PRD-xxxx.md`를 생성한다.

### 3단계: 셀프 체크

- [ ] 필수 섹션(배경, 목표/성공 기준, 사용자 스토리, 범위, 제약사항, 변경 이력)이 모두 포함되었는가?
- [ ] 메타데이터(문서 ID, 버전, 상태, 오너)가 정확한가?
- [ ] 범위와 비범위가 명확히 구분되었는가?
- [ ] 비기능 요구사항(NFR)이 정량적으로 기술되었는가?
- [ ] 사용자 스토리가 한글로 작성되었는가?
- [ ] 변경 이력이 기록되었는가?

### 4단계: 보고

```
## PRD 작성 보고서
- 생성 문서: PRD-xxxx
- 발견 사항: (모호하거나 추가 확인이 필요한 부분)
- 오픈 이슈: (해결되지 않은 질문)
- 판단 근거: (주요 결정과 근거)
```

## 작성 규칙

- 사용자 스토리는 한글로 작성: `[역할]로서, [기능]을 원한다. 그래야 [가치].`
- 기술적 구현 상세(기술 스택, 저장소, 이벤트 인프라 등)는 부록으로 분리한다.
- 도메인 규칙/계약은 본문에 포함한다.
- 승인되지 않은 가정이 필요하면 오픈 이슈로 올린다.
- PRD 범위를 벗어나는 내용은 `CHANGE_REQUEST`로 보고한다.
