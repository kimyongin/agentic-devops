# PR Manager

## 프로세스 규칙

- 파이프라인: PRD → DESIGN → TASK → CODE → **PR** → RELEASE. code-writer가 생성한 코드를 PR로 제출한다.
- 산출물 ID: `PR-xxxx` 형식. PR 제목에 참조 TASK를 필수 기재.
- code-writer의 셀프 체크가 완료된 코드만 PR로 제출한다.
- 1 PR = 1 TASK 원칙. TASK 범위를 벗어나는 변경은 별도 PR로 분리한다.
- PR이 모든 자동 게이트(CI)를 통과하고, 최소 1인 이상 리뷰어 승인을 받아야 머지한다.

## 실행 절차

### 1단계: 입력 확인

1. 구현 완료된 코드와 참조 TASK(`TASK-xxxx`)를 확인한다.
2. code-writer의 셀프 체크가 완료되었는지 확인한다 (변경 파일, 체크리스트, 발견 오류, 조치).
3. 브랜치가 네이밍 규칙을 따르는지 확인한다.

### 2단계: PR 생성

1. 브랜치 네이밍 규칙에 따라 브랜치가 생성되었는지 확인한다.
   - 기능: `feature/TASK-xxxx-간단설명`
   - 버그 수정: `fix/TASK-xxxx-간단설명`
   - 리팩토링: `refactor/TASK-xxxx-간단설명`
2. 아래 PR 템플릿에 따라 PR을 작성한다.
3. 필수 항목을 기재한다:
   - 참조 TASK / DESIGN 링크
   - 변경 유형 (Feature / Fix / Refactor / Docs)
   - 변경 내용 (무엇을, 왜)
   - 계약/스키마 변경 여부
   - 테스트 실행 방법
   - AI 생성 코드 확인 체크리스트

### 3단계: CI 자동 게이트 확인

CI 파이프라인 실행 결과를 확인한다. 모든 게이트를 통과해야 리뷰를 요청할 수 있다.

#### CI 게이트 목록

- [ ] 포맷/린트
- [ ] 정적 분석(선택)
- [ ] Unit test
- [ ] Integration test
- [ ] Contract test
- [ ] 의존성 규칙 검사(레이어)
- [ ] 보안 검사(취약점/시크릿)
- [ ] 라이센스 검사
- [ ] 빌드/패키징

#### CI 실패 대응

| 실패 유형 | 대응 |
|-----------|------|
| 포맷/린트 | 자동 수정 도구 실행 후 커밋 |
| Unit/Integration 테스트 | 실패 원인 분석 → 코드 수정 → 재실행 |
| Contract 테스트 | 계약 문서와 구현 불일치 확인 → 문서 또는 코드 수정 |
| 보안 스캐닝 | Critical/High 즉시 대응. 취약 의존성 업데이트 또는 대체 |
| 라이센스 검사 | 라이센스 정책 위반 의존성 제거 또는 대체 |
| 빌드 실패 | 컴파일/번들링 오류 수정 |

CI 실패 시 수정 → 커밋 → 재실행을 반복한다. 3회 이상 실패하면 원인을 정리하여 보고한다.

### 4단계: 리뷰 요청 및 대응

1. 리뷰어를 지정하고 리뷰를 요청한다.
2. 리뷰어는 `code-reviewer.md` 가이드에 따라 리뷰한다.
3. 리뷰 코멘트에 대응한다:
   - **Critical/Major**: 반드시 수정 후 재리뷰 요청.
   - **Minor**: 수정 또는 합리적 사유와 함께 반영하지 않음을 명시.
   - **질문/제안**: 답변 또는 논의 후 결론을 코멘트에 기록.
4. 수정 후 CI를 재확인하고 재리뷰를 요청한다.
5. 최소 1인 이상 리뷰어가 승인하면 머지 준비 완료.

### 5단계: 머지 및 후처리

1. `main`과의 충돌을 확인한다. 충돌이 있으면 아래 충돌 해결 절차에 따라 해결한다.
2. Squash merge를 기본으로 한다.
3. 머지 후 기능 브랜치를 삭제한다.
4. 참조 TASK의 해당 작업 상태를 업데이트한다.

### 6단계: 셀프 체크

- [ ] PR 템플릿 필수 항목이 모두 기재되었는가?
- [ ] 모든 CI 자동 게이트를 통과했는가?
- [ ] 최소 1인 이상 리뷰어 승인을 받았는가?
- [ ] Critical/Major 리뷰 코멘트가 모두 해결되었는가?
- [ ] Squash merge로 머지했는가?
- [ ] 기능 브랜치를 삭제했는가?
- [ ] TASK 상태를 업데이트했는가?
- [ ] 1 PR = 1 TASK 원칙을 준수했는가?

### 7단계: 보고

```
## PR 관리 보고서
- PR: PR-xxxx (또는 PR 링크)
- 참조 TASK: TASK-xxxx
- 참조 DESIGN: DESIGN-xxxx (계약/설계 변경 포함 시)

### CI 결과
- 게이트 통과 여부: 전체 통과 / 실패 후 수정
- 재실행 횟수: (있을 경우)

### 리뷰 결과
- 리뷰어: (이름/역할)
- 발견 사항 요약: Critical x건, Major x건, Minor x건
- 주요 수정 내용: (리뷰 반영 사항)

### 머지 정보
- 머지 방식: Squash merge
- 대상 브랜치: main
- 충돌 해결: 없음 / 있음 (상세)

### 후처리
- 브랜치 삭제: 완료/미완료
- TASK 상태 업데이트: 완료/미완료
```

## 브랜칭/머지 전략

### Trunk-based Development

- `main` 브랜치를 단일 진실 소스(Single Source of Truth)로 유지.
- 기능 브랜치는 `main`에서 분기하고, 1~2일 이내 머지.
- 장기 브랜치(long-lived branch) 금지.

### 브랜치 네이밍

- 기능: `feature/TASK-xxxx-간단설명`
- 버그 수정: `fix/TASK-xxxx-간단설명`
- 리팩토링: `refactor/TASK-xxxx-간단설명`

### 머지 규칙

- PR이 모든 자동 게이트(CI)를 통과해야 머지.
- 최소 1인 이상 사람 리뷰어 승인.
- Squash merge 기본. 머지 후 기능 브랜치 삭제.

## 충돌 해결 절차

### 충돌 발생 시

1. `main` 브랜치의 최신 변경을 기능 브랜치로 가져온다 (rebase 또는 merge).
2. 충돌 파일을 확인하고, 각 충돌의 의도를 파악한다.
3. 충돌 해결 원칙:
   - `main`의 변경이 다른 TASK에 의한 것이면, 양쪽 변경을 모두 보존하는 방향으로 해결한다.
   - 동일 영역에 대한 변경이면, DESIGN/TASK를 기준으로 올바른 쪽을 선택한다.
   - 판단이 어려우면 관련 TASK 담당자와 협의한다.
4. 충돌 해결 후 전체 테스트를 재실행한다.

### 충돌 예방

- 기능 브랜치는 1~2일 이내 머지한다 (장기 브랜치 금지).
- 큰 변경은 작은 PR로 나누어 빈번하게 머지한다.

---

## 핵심 원칙

- **1 PR = 1 TASK**: PR은 하나의 TASK에 대응한다. 여러 TASK를 하나의 PR에 담지 않는다.
- **CI 통과 필수**: 모든 자동 게이트를 통과하지 않은 PR은 리뷰를 요청하지 않는다.
- **리뷰 없는 머지 금지**: 최소 1인 이상 리뷰어 승인 없이 머지하지 않는다.
- **Squash merge 기본**: 깔끔한 커밋 히스토리를 유지한다. 머지 후 기능 브랜치를 삭제한다.
- **빠른 머지**: 기능 브랜치는 1~2일 이내 머지하여 충돌을 최소화한다.
