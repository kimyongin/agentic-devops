# PRD Writer

> 파이프라인 규칙, 산출물 체계, 역할, 변경 처리는 `SKILL.md`를 따른다.

- PRD는 파이프라인의 첫 단계 산출물이다. 산출물 ID: `PRD-xxxx-slug`.

## 목차
- [실행 절차](#실행-절차)
- [작성 규칙](#작성-규칙)
- [PRD 템플릿](#prd-템플릿)

## 실행 절차

prd-writer는 두 가지 모드로 동작한다. 오케스트레이터가 스폰 시 어떤 모드인지 명시한다.

---

### 모드 1: 템플릿 생성 모드

오케스트레이터로부터 요구사항과 함께 "템플릿 생성" 지시를 받았을 때 실행한다.

#### 1단계: 순번 결정

`docs/ai-devops/PRD/` 디렉터리의 기존 파일을 스캔하여 마지막 순번 + 1을 사용한다. 파일이 없으면 `0001`부터 시작한다.

#### 2단계: PRD 템플릿 파일 생성

요구사항을 참고하여 각 섹션에 작성 가이드와 예시를 포함한 템플릿 파일을 생성한다.

- 파일 경로: `docs/ai-devops/PRD/PRD-xxxx-slug-TEMPLATE.md`
- AI가 판단할 수 있는 내용은 초안으로 채워 넣고, 사용자만이 결정할 수 있는 항목은 `<!-- TODO: 작성 필요 -->` 표시를 남긴다.
- 기술 스택 섹션은 사용자가 직접 입력해야 하므로 반드시 빈 칸으로 둔다.

#### 3단계: 보고

```
## PRD 템플릿 생성 보고서
- 생성 파일: docs/ai-devops/PRD/PRD-xxxx-slug-TEMPLATE.md
- AI 초안 작성 항목: (AI가 채워넣은 섹션 목록)
- 사용자 입력 필요 항목: (TODO로 남긴 항목 목록, 특히 기술 스택)
- 안내: 템플릿을 작성한 후 "PRD 검토 요청"을 전달하면 최종 PRD로 확정합니다.
```

---

### 모드 2: 검토 모드

오케스트레이터로부터 작성 완료된 템플릿 파일 경로와 함께 "검토" 지시를 받았을 때 실행한다.

#### 1단계: 작성된 템플릿 확인

1. 지정된 템플릿 파일(`PRD-xxxx-slug-TEMPLATE.md`)을 읽는다.
2. `<!-- TODO -->` 항목이 모두 작성되었는지 확인한다. 미작성 필수 항목이 있으면 사용자에게 보고하고 검토를 중단한다.

#### 2단계: 내용 검토 및 정제

아래 **작성 규칙**을 기준으로 검토한다:

- 모호한 요구사항이 있으면 명확화하거나 오픈 이슈로 기록한다.
- MoSCoW 우선순위가 적절한지 확인하고 조정안을 제안한다.
- NFR(비기능 요구사항)이 정량적으로 기술되었는지 확인한다.
- 범위/비범위가 명확히 구분되었는지 확인한다.
- 기술 스택 섹션이 작성되었는지 확인한다 (DESIGN 단계의 입력으로 필수).

#### 3단계: 최종 PRD 저장

검토 완료 후 최종 PRD를 저장한다.

- 파일 경로: `docs/ai-devops/PRD/PRD-xxxx-slug.md`
- 상태: `DRAFT`

#### 4단계: 보고

```
## PRD 검토 보고서
- 확정 문서: PRD-xxxx-slug
- 주요 검토 사항: (검토 중 수정하거나 명확화한 내용)
- 발견 사항: (모호하거나 추가 확인이 필요한 부분)
- 오픈 이슈: (해결되지 않은 질문)
- 잔여 리스크: (남아 있는 위험 요소)
```

---

## 작성 규칙

- 이해관계자(Stakeholder)를 식별하고, 누구의 요구를 반영하는지 명시한다.
- 기능 우선순위는 MoSCoW 기준으로 분류한다:
  - **Must**: 반드시 필요. 없으면 릴리스 불가.
  - **Should**: 중요하지만 없어도 릴리스 가능. 차선책이 존재.
  - **Could**: 있으면 좋지만 일정에 따라 제외 가능.
  - **Won't (this time)**: 이번 범위에서 명시적으로 제외. 비범위에 기록.
- 상충하는 요구사항이 있으면 이해관계자와 우선순위를 근거로 판단하고, 판단이 어려우면 오픈 이슈로 올린다.
- 사용자 스토리는 한글로 작성: `[역할]로서, [기능]을 원한다. 그래야 [가치].`
- 도메인 규칙/계약은 본문에 포함한다.
- 승인되지 않은 가정이 필요하면 오픈 이슈로 올린다.
- PRD 범위를 벗어나는 내용은 `CHANGE_REQUEST`로 보고한다.
- **기술 스택은 사용자가 직접 결정한다.** AI는 기술 스택 섹션을 임의로 채우지 않는다. DESIGN 단계에서 이 정보를 기반으로 아키텍처를 설계한다.
