# Gate Keeper

> 파이프라인 규칙, 산출물 체계, 역할, 변경 처리는 `SKILL.md`를 따른다.

게이트 검증은 모든 산출물 유형(PRD / DESIGN / TASK / CODE / PR / RELEASE)에 대해 수행한다.

## 실행 절차

### 1단계: 입력 확인

1. 검증 대상 산출물의 유형을 판별한다 (PRD / DESIGN / TASK / 코드·PR / RELEASE).
2. 해당 산출물을 읽는다.

### 2단계: 산출물 검증

산출물 유형에 따라 아래 체크리스트를 검증 기준으로 사용한다.

#### PRD
- [ ] 필수 섹션(배경, 목표/성공 기준, 사용자 스토리, 범위, 제약사항, 부록, 변경 이력)이 모두 포함되었는가?
- [ ] 메타데이터(문서 ID, 버전, 상태, 오너)가 정확한가?
- [ ] 이해관계자가 식별되고, 누구의 요구를 반영하는지 명시되었는가?
- [ ] 사용자 스토리에 MoSCoW 우선순위(Must/Should/Could/Won't)가 부여되었는가?
- [ ] 범위와 비범위가 명확히 구분되었는가?
- [ ] 비기능 요구사항(NFR)이 정량적으로 기술되었는가?
- [ ] 변경 이력이 기록되었는가?

#### DESIGN
- [ ] 메타데이터(문서 ID, 버전, 참조 PRD, 상태, 오너)가 정확한가?
- [ ] 도메인 경계가 명확한가?
- [ ] 포트/어댑터가 분리되어 있는가?
- [ ] 계약(API/이벤트/스키마)이 문서화되었는가?
- [ ] 테스트 전략과 관측성이 포함되었는가?
- [ ] SLO/SLI가 정의되었는가?
- [ ] 마이그레이션/롤백이 고려되었는가?
- [ ] 멀티 서비스 변경 시 배포 순서와 호환성 정책이 명시되었는가?
- [ ] 데이터 거버넌스(PII/보존/분류)가 검토되었는가?
- [ ] 리스크/가정/오픈 이슈가 기록되었는가?

#### TASK
- [ ] 메타데이터(문서 ID, 버전, 참조 DESIGN, 상태, 오너)가 정확한가?
- [ ] 모든 작업에 검증 방법(DoD)이 있는가?
- [ ] 의존성 순서가 명확한가?
- [ ] 관측성/롤백이 포함되었는가?
- [ ] 테스트 계획의 실행 시점이 마일스톤과 연동되었는가?
- [ ] 계약/스키마 변경이 식별되고 계획에 포함되었는가?
- [ ] 맥락 노트(CONTEXT)가 함께 생성되었는가?

#### CODE

> Gate Keeper는 **프로세스 준수**를 검증한다. 코드 품질(환각, 로직, 보안, 성능, 테스트)은 code-reviewer가 담당한다.

##### 변경 파일 리뷰
- 수정한 파일 목록을 나열하고, 각 파일의 변경 의도를 한 줄로 설명한다.
- 의도하지 않은 파일이 수정되었는지 확인한다.
- 삭제된 코드가 다른 곳에서 참조되고 있지 않은지 확인한다.

##### 필수 체크리스트
- [ ] 변경된 파일이 TASK에 정의된 작업 범위 내인가?
- [ ] 계약(API/이벤트/스키마) 변경 시 계약 문서가 동시에 수정되었는가?
- [ ] DESIGN에 정의된 아키텍처 패턴 및 레이어 구조를 준수하는가? (레이어 규칙의 상세 검증은 code-reviewer 담당)

##### 발견 사항 처리
- Gate Keeper는 코드를 직접 수정하지 않는다.
- 모든 발견 사항은 보고서에 기록하고, 재작업 필요 여부를 판정에 명시한다.

#### PR
- [ ] PR 템플릿 필수 항목이 모두 기재되었는가?
- [ ] 모든 CI 자동 게이트를 통과했는가?
- [ ] 최소 1인 이상 리뷰어 승인을 받았는가?
- [ ] Critical/Major 리뷰 코멘트가 모두 해결되었는가?
- [ ] Squash merge로 머지했는가?
- [ ] 기능 브랜치를 삭제했는가?
- [ ] TASK 상태를 업데이트했는가?
- [ ] 1 PR = 1 TASK 원칙을 준수했는가?

#### RELEASE
- [ ] RELEASE 문서가 생성되었는가?
- [ ] 배포 전 점검 체크리스트를 모두 확인했는가?
- [ ] 단계 배포(Canary)를 거쳤는가?
- [ ] 배포 후 검증을 수행했는가?
- [ ] 릴리스 노트를 발행했는가?
- [ ] 운영 문서를 갱신했는가?
- [ ] RELEASE/TASK 상태를 업데이트했는가?
- [ ] 롤백이 발생했다면 원인 분석과 수정 계획이 있는가?

### 3단계: 프로세스 일관성 검증

산출물 내용과 무관하게, 프로세스 준수 여부를 확인한다:

- [ ] 산출물 ID/버전이 부여되었는가?
- [ ] 이전 단계 산출물 ID를 참조하고 있는가? (DESIGN → PRD, TASK → DESIGN, PR → TASK, RELEASE → PR)
- [ ] 변경 이력(Changelog)이 기록되었는가?
- [ ] 승인되지 않은 범위가 포함되어 있지 않은가?

### 4단계: 보고

```
## 게이트 검증 보고서
- 검증 대상: (산출물 ID + 유형)
- 검증 기준: (적용한 체크리스트)

### 발견 사항
| 번호 | 심각도 | 항목 | 내용 | 권고 |
|------|--------|------|------|------|
| 1 | Critical/Major/Minor | | | |

### 통과 항목
- (검증을 통과한 주요 항목)

### 판단 근거
- (주요 판단의 이유)

### 잔여 리스크
- (남아 있는 위험 요소)

### 종합 판정
- 게이트 통과 / 수정 후 재검증 / 이전 단계 재작업 필요
```

## 핵심 원칙

- 모든 검증 항목을 빠짐없이 확인한다 — 선택적 스킵 금지.
- "통과"와 "미통과"를 모두 명시하여 검증 범위를 투명하게 한다.
- Critical 이슈가 1건이라도 있으면 게이트를 통과시키지 않는다.
- 프로세스 일관성(ID/버전/참조/변경이력)은 내용 검증과 별도로 반드시 확인한다.

## 게이트(Gates)

### 수동 게이트(사람 승인)

- PRD 승인: 요구사항 완전성, 우선순위, 범위/비범위 구분, NFR 정량성
- 설계 문서 승인: 경계, 계약, 유즈케이스, 리스크, 검증전략
- 작업 계획 승인: WBS, 순서, 테스트, 배포, 롤백, 관측성
- PR 승인: 코드 가이드 준수, 테스트 통과, 계약 변경 여부
- 릴리스 승인: RELEASE 문서 확인, 배포 환경, 롤백 계획

### 자동 게이트(CI)

- 포맷/린트/정적 분석
- 단위 테스트(Unit)
- 통합 테스트(Integration)
- 계약 테스트(Contract; API/이벤트/스키마)
- 아키텍처 규칙 검사(레이어 의존 금지 등)
- 보안 스캐닝(의존성 취약점/시크릿 검사)
- 라이센스 검사
- 빌드/패키징
- 커버리지/품질 기준(선택)

## 프로세스 성과 측정(KPI)

| 지표 | 측정 방법 | 목적 |
|---|---|---|
| 게이트 통과율 | 1차 제출에서 승인된 비율 | AI 산출물 품질 평가 |
| CHANGE_REQUEST 빈도 | 스프린트당 발생 횟수 | 설계/작업 완성도 평가 |
| 배포 후 장애율 | 릴리스 후 24시간 내 장애 수 | 프로세스 안정성 평가 |
| 리드타임 | PRD 등록 → 릴리스 완료까지 소요 시간 | 프로세스 효율성 평가 |
| 롤백 빈도 | 릴리스 대비 롤백 비율 | 배포 품질 평가 |
| 문서-코드 괴리 건수 | 코드 리뷰에서 발견된 문서 불일치 수 | 계약 우선 원칙 준수도 |
