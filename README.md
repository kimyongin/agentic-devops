# AgenticDevOps

AI 에이전트와 함께 소프트웨어를 설계하고, 구현하고, 배포하기 위한 스킬.

## 왜 만들었나

AI 에이전트는 코드를 잘 작성하지만, 혼자 놔두면 설계 없이 코드를 짜고, 범위를 넘어서 변경하고, 문서 없이 배포한다. 사람이 매번 "설계 먼저 해", "계약 문서 갱신해", "테스트 빠졌어"를 지시하는 건 비효율적이다.

이 스킬은 AI 에이전트에게 **개발 프로세스 자체를 가르쳐서**, 사람은 의사결정과 승인에 집중하고 AI가 프로세스를 스스로 따르게 한다.

## 설계 컨셉

### 산출물 기반 파이프라인

모든 개발을 6단계 파이프라인으로 구조화한다. 각 단계는 이전 단계의 **승인된 산출물**을 입력으로 받으며, 승인 없이 다음 단계로 넘어갈 수 없다.

```
PRD → DESIGN → TASK → CODE → PR → RELEASE
```

- **PRD**: 무엇을 만들 것인가 (요구사항)
- **DESIGN**: 어떻게 만들 것인가 (아키텍처, 계약, 테스트 전략)
- **TASK**: 어떤 순서로 만들 것인가 (WBS, 마일스톤)
- **CODE → PR**: 구현하고 머지한다
- **RELEASE**: 배포하고 검증한다

### AI가 쓰고, AI가 검증하고, 사람이 승인한다

AI Writer가 산출물을 생성하면, Gate Keeper(역시 AI)가 체크리스트로 검증하고 보고서를 작성한다. 사람은 보고서를 참고하여 승인/반려만 하면 된다. 사람이 세부 체크를 할 필요 없이, AI끼리 교차 검증한 결과를 판단하는 구조다.

### 작업 기억 프로토콜

AI 에이전트는 대화가 길어지면 맥락을 잃는다. 이 스킬은 **TASK(계획) + CONTEXT(결정 근거) + WBS 상태(진행 체크리스트)** 3종 문서를 운영하여, 새 대화에서도 이전 맥락을 이어갈 수 있게 한다.

### 변경은 추적하고, 범위는 통제한다

모든 산출물은 ID/버전으로 추적되며, 이전 단계 산출물을 참조한다. 승인 범위를 벗어나는 변경이 필요하면 AI가 자동으로 `CHANGE_REQUEST`를 발행하고, 해당 단계의 문서를 갱신한 뒤 재승인을 받는다.

## 기술 스택

| 영역 | 아키텍처 | 비고 |
|------|----------|------|
| 백엔드 | 헥사고날 아키텍처 (Ports & Adapters) | 도메인 순수성, 모듈 경계 분리 |
| 프론트엔드 | Feature-Sliced Design (FSD) | 레이어 기반, 의존 방향 강제 |
| 인프라 | Kubernetes | 매니페스트/Helm, IaC |
| 배포 단위 | 모듈러 모놀리스 (기본) → 모노레포 MSA (필요 시) | 헥사고날 포트 교체로 자연 전환 |

## 설치

```bash
npx skills add <user>/ai-driven-devops-skills
```

Cursor, Claude Code 등 Agent Skills 표준을 지원하는 AI 도구에 자동 설치된다.

## 사용법

개발 관련 요청을 하면 스킬이 자동으로 활성화된다. 파이프라인 순서에 따라 자연어로 요청하면 된다.

### 신규 프로젝트

```
유저: "온라인 쇼핑몰을 만들고 싶어. PRD 작성해줘"
  → AI가 PRD-0001-shopping-mall.md 생성 + Gate Keeper 검증 보고서
유저: "승인"

유저: "인증 도메인부터 설계해줘"
  → AI가 DESIGN-0001-auth-domain.md 생성 + 검증 보고서
유저: "승인"

유저: "로그인 API부터 작업 계획 작성해줘"
  → AI가 TASK-0001-login-api.md + CONTEXT 생성 + 검증 보고서
유저: "승인"

유저: "구현해줘"
  → 코드/테스트 생성
유저: "PR 만들어줘"
  → PR 생성, CI 확인, 리뷰 조율
유저: "머지 승인"

유저: "릴리스해줘"
  → RELEASE-0001-v1.0.0.md 생성 + 검증 보고서
유저: "승인" → 배포
```

### 기존 기능 변경

```
유저: "인증에 OAuth를 추가하고 싶어"
  → AI가 DESIGN을 DRAFT로 되돌리고 갱신 + 검증 보고서
유저: "승인" → TASK → CODE → PR → RELEASE
```

### 개발 중 범위 변경

```
유저: "구현해줘"
  → AI: "새 API 엔드포인트가 필요합니다. CHANGE_REQUEST를 발행합니다."
  → DESIGN 갱신 → 승인 → TASK 갱신 → 구현 재개
```

## 구조

```
skills/agentic-devops/
├── SKILL.md                         # 파이프라인 정의 + 작업별 라우팅 테이블
└── references/
    ├── prd-writer.md                # PRD 작성 가이드
    ├── design-writer.md             # 설계 문서 가이드
    ├── task-writer.md               # 작업 계획 가이드
    ├── code-writer.md               # 코드 생성 / PR 관리 가이드
    ├── code-reviewer.md             # AI 생성 코드 리뷰 가이드
    ├── release-manager.md           # 릴리스/배포/롤백 가이드
    └── gate-keeper.md               # 파이프라인 게이트 검증 가이드
```

- `SKILL.md`는 AI 에이전트가 항상 먼저 읽는 진입점이다. 핵심 원칙과 라우팅 테이블을 담고 있다.
- `references/`의 각 파일은 요청에 해당하는 것만 선택적으로 로드된다. 컨텍스트 윈도우를 절약하기 위해 필요한 가이드만 읽는 구조다.
